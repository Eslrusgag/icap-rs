# RFC 3507 Support Matrix

This document summarizes how `icap-rs` maps to RFC 3507 behavior. The executable
conformance matrix lives in `icap-rs/tests/rfc3507.rs`.

Tests named `supported_*` are release assertions. Ignored tests named
`unsupported_*` document visible gaps so unsupported behavior is not mistaken
for implemented behavior.

## Supported

- ICAP/1.0 request line and response status line handling.
- `OPTIONS`, `REQMOD`, and `RESPMOD`.
- Required `Host` validation on incoming requests.
- `Encapsulated` parsing and serialization.
- Method-specific `Encapsulated` validation for ICAP requests.
- Rejection of duplicate, invalid, non-monotonic, and incompatible
  `Encapsulated` forms.
- Embedded HTTP request and response support.
- RFC framing for embedded HTTP:
  - embedded HTTP heads are written unchunked;
  - ICAP chunked coding starts at `req-body`, `res-body`, or `opt-body`;
  - response parsing dechunks encapsulated entity bodies before returning
    `Response::body`.
- Client response header parsing treats `CRLFCRLF` or EOF as the response
  header boundary. A 4xx/5xx status line alone does not terminate the ICAP
  response while the connection remains open.
- Preview flow:
  - `Preview: 0`;
  - `Preview: N`;
  - `ieof`;
  - `100 Continue`;
  - preview-aware handlers that can return a final response before
    `100 Continue`.
- `204 No Content` validation and serialization with
  `Encapsulated: null-body=0`.
- Server guard for `204`: without `Allow: 204` and without Preview, a handler
  `204` is converted to `200 OK` with the embedded HTTP message echoed.
- `206 Partial Content` no-modification support through the
  `use-original-body` marker.
- `ISTag` validation for successful ICAP responses. Outgoing responses are
  normalized to the RFC 3507 quoted-string syntax (`ISTag: "..."`). The parser
  also accepts unquoted token/base64-like `ISTag` values as explicit
  compatibility behavior for existing ICAP implementations.
- Server `ServiceOptions` requires an explicit static or dynamic `ISTag`; no
  default tag is generated for routes or automatic `OPTIONS` responses.
- `OPTIONS` capability responses:
  - `Methods`;
  - `Service`;
  - `Service-ID`;
  - `ISTag`;
  - `Options-TTL`;
  - `Max-Connections`;
  - `Max-Object-Size`;
  - `Allow`;
  - `Preview`;
  - `Transfer-Preview`;
  - `Transfer-Ignore`;
  - `Transfer-Complete`;
  - optional `opt-body`.
- Per-service embedded HTTP object-size limits with
  `ServiceOptions::with_max_object_size`: `OPTIONS` advertises
  `Max-Object-Size`, and request handling enforces the configured byte limit
  by counting decoded ICAP chunked body bytes rather than trusting embedded
  HTTP `Content-Length`.
- Keep-alive connections without pipelining.
- Client-side OPTIONS response caching with `Options-TTL` lifetime and `ISTag`
  invalidation (RFC 3507 §4.10 / §5), opt-in via
  `ClientBuilder::with_options_cache`.
- Client-side `Transfer-Preview` / `Transfer-Ignore` / `Transfer-Complete`
  policy resolved per file extension from the cached `OPTIONS` response
  (RFC 3507 §4.10.2): ignore returns a synthetic `204`, complete sends the full
  body with no `Preview` header, preview sends the advertised window before
  `100 Continue`.
- Client `Proxy-Authorization: Basic` retry on `407 Proxy Authentication
  Required` (RFC 3507 §7.1), opt-in via `ClientBuilder::proxy_auth`.
- Direct ICAPS (`icaps://`) through Rustls when `tls-rustls` is enabled.
- Rustls server TLS and mTLS listeners.

## Explicit Compatibility Behavior

- Strict request parsing is the default.
- Legacy `OPTIONS` requests without `Encapsulated` are rejected by strict mode.
- Servers can opt into compatibility parsing with
  `ServerBuilder::with_compatibility_request_parser`.
- `ClientTlsConfig::dangerous_disable_cert_verification` and the CLI
  `--insecure` flag actually disable Rustls server certificate verification
  (equivalent to `c-icap-client -tls-no-verify`). They are intentionally
  named `dangerous_` so code review can flag misuse; do not enable in
  production.
- Incoming unquoted token/base64-like `ISTag` values are accepted for
  interoperability. This compatibility does not change generated wire output:
  `Response` and `ServiceOptions` emit quoted `ISTag` values.
- On EOF after a status-line-only 4xx/5xx response, the client normalizes the
  missing header terminator as a legacy compatibility behavior. The same
  shortcut is intentionally not applied while the TCP connection is still open.

Compatibility behavior should remain explicit, documented, and tested.

## Partial Support

- The client `Transfer-*` policy and OPTIONS cache are opt-in: with no
  `ClientBuilder::with_options_cache` configured, the client never fetches
  `OPTIONS` automatically and applies no transfer policy. Advanced RFC cache
  semantics beyond `Options-TTL` lifetime and `ISTag` invalidation are not
  modeled.
- Server-side ICAP authentication enforcement is not implemented: the client
  sends `Proxy-Authorization` on `407` (RFC 3507 §7.1), but the server does not
  challenge or verify credentials.
- Service routing uses the resolved service path segment rather than the full
  RFC service URI identity model.
- Chunk extensions used by supported flows are parsed, including `ieof` and
  `use-original-body`, but trailer headers do not have a structured public API.
