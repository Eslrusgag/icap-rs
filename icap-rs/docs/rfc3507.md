# RFC 3507 Support Matrix

This document summarizes how `icap-rs` maps to RFC 3507 behavior. The executable
conformance matrix lives in `icap-rs/tests/rfc3507.rs`.

Tests named `supported_*` are release assertions. Ignored tests named
`unsupported_*` document visible gaps so unsupported behavior is not mistaken
for implemented behavior.

## Supported

- ICAP/1.0 request line and response status line handling.
- `OPTIONS`, `REQMOD`, and `RESPMOD`.
- Required `Host` validation on incoming requests.
- `Encapsulated` parsing and serialization.
- Method-specific `Encapsulated` validation for ICAP requests.
- Rejection of duplicate, invalid, non-monotonic, and incompatible
  `Encapsulated` forms.
- Embedded HTTP request and response support.
- RFC framing for embedded HTTP:
  - embedded HTTP heads are written unchunked;
  - ICAP chunked coding starts at `req-body`, `res-body`, or `opt-body`;
  - response parsing dechunks encapsulated entity bodies before returning
    `Response::body`.
- Preview flow:
  - `Preview: 0`;
  - `Preview: N`;
  - `ieof`;
  - `100 Continue`;
  - preview-aware handlers that can return a final response before
    `100 Continue`.
- `204 No Content` validation and serialization with
  `Encapsulated: null-body=0`.
- Server guard for `204`: without `Allow: 204` and without Preview, a handler
  `204` is converted to `200 OK` with the embedded HTTP message echoed.
- `206 Partial Content` no-modification support through the
  `use-original-body` marker.
- `ISTag` validation for successful ICAP responses.
- `OPTIONS` capability responses:
  - `Methods`;
  - `Service`;
  - `Service-ID`;
  - `ISTag`;
  - `Options-TTL`;
  - `Max-Connections`;
  - `Allow`;
  - `Preview`;
  - `Transfer-Preview`;
  - `Transfer-Ignore`;
  - `Transfer-Complete`;
  - optional `opt-body`.
- Keep-alive connections without pipelining.
- Direct ICAPS (`icaps://`) through Rustls when `tls-rustls` is enabled.
- Rustls server TLS and mTLS listeners.

## Explicit Compatibility Behavior

- Strict request parsing is the default.
- Legacy `OPTIONS` requests without `Encapsulated` are rejected by strict mode.
- Servers can opt into compatibility parsing with
  `ServerBuilder::with_compatibility_request_parser`.
- `ClientBuilder::insecure_no_verify` and the CLI `--insecure` flag are kept as
  compatibility surfaces, but they do not disable Rustls certificate
  verification.

Compatibility behavior should remain explicit, documented, and tested.

## Partial Support

- `Transfer-Preview`, `Transfer-Ignore`, and `Transfer-Complete` can be
  advertised by `ServiceOptions`, but the client does not automatically consume
  the full transfer policy model.
- Cache-related headers can be represented, but the client does not implement a
  complete RFC OPTIONS cache and invalidation model.
- Service routing uses the resolved service path segment rather than the full
  RFC service URI identity model.
- Chunk extensions used by supported flows are parsed, including `ieof` and
  `use-original-body`, but trailer headers do not have a structured public API.
